# OpenClaw セキュリティリスク評価書

> 最終更新: 2026-03-23
> ステータス: 導入前リスク評価

## 結論（先に）

**現時点での本番導入は非推奨**。セキュリティ面の懸念が多すぎる。導入する場合は完全隔離環境（Docker/VM）+ 最小権限が必須。

---

## 1. 重大脆弱性: ClawJacked（CVE-2026-25253）

| 項目 | 内容 |
|------|------|
| 深刻度 | CVSS 8.8（高） |
| 種別 | 認証トークン窃取 + エージェント完全乗っ取り |
| 修正済み | v2026.2.25 以降で修正 |

### 攻撃の流れ

1. 被害者が攻撃者のWebサイトを訪問（普通のブラウジング）
2. JavaScriptがlocalhost上のOpenClawゲートウェイにWebSocket接続
3. レートリミッターが**localhost接続を免除**しているため、パスワードを総当たり
4. 認証成功後、自動でデバイスペアリング（**ユーザーへの確認プロンプトなし**）
5. **エージェント完全掌握**: APIキー・セッションtokenの窃取、コマンド実行

### 影響

AIエージェントがAPIキー、ファイルシステムアクセス、各種ツール実行権限を持っているため、乗っ取られた場合の被害範囲が極めて広い。

## 3. サプライチェーンリスク

### ClawHub マーケットプレイス汚染

- **ClawHavoc キャンペーン**: 1,184件の悪意あるスキルが発見（全体の約20%）
- 主な攻撃: Atomic macOS Stealer（AMOS）— ブラウザ認証情報・暗号通貨ウォレット・APIキーの窃取
- Snyk の監査: **全ClawHubスキルの36%にプロンプトインジェクションを検出**

### 開発者の離脱

- 創設者 Peter Steinberger が2026年2月にOpenAIに移籍
- プロジェクトの今後のガバナンス・セキュリティ対応力に懸念

### 各国政府の対応

- 中国: 政府機関・国有企業でのOpenClaw使用を制限
- 組織の18%のみがIAMシステムでAIエージェントに対応できると回答

## 4. データプライバシーリスク

### APIキー漏洩

- エラーメッセージからAPIキーが監視ダッシュボードに漏洩するケース
- 設定ファイルに平文でAPIキーが保存される設計
- 研究者が認証なし公開インスタンスからAnthropic APIキー・Telegramトークン・Slackアカウントを抽出

### ローカルファイルアクセス

- Docker隔離なしの場合、環境変数・設定ファイル・全ファイルシステムにアクセス可能
- 侵害されたエージェントは実行ユーザーの全ディスク権限を継承

### 永続メモリ攻撃

- `MEMORY.md`、`SOUL.md` への注入で「遅延型攻撃」が可能
- 注入された指示が数週間後に発動するケースあり

## 5. Ryo の事業への影響評価

### リスクが高い理由

| 項目 | Ryo の状況 | リスク |
|------|-----------|--------|
| APIキー | Stripe、Google API、LINE等を多数利用 | 漏洩すると決済・顧客情報に直結 |
| 顧客データ | Airbnbゲスト情報、えんがわ予約データ | 個人情報保護法違反のリスク |
| 財務情報 | 経費管理、売上データ | 事業機密の漏洩 |
| メッセージング連携 | LINE、WhatsApp等 | なりすまし・情報漏洩 |

### もし導入するなら（最低限の対策）

1. **Docker コンテナ内で実行**（ホストOSから完全隔離）
2. **専用の制限付きユーザー**で実行（root禁止）
3. **ポート18789を外部に公開しない**（localhost のみ or Tailscale経由）
4. **v2026.2.26 以上を使用**（ClawJacked修正済み）
5. **サンドボックスモードを有効化**（デフォルトはOFF）
6. **ClawHubからのスキルは使わない**（全体の36%にインジェクション検出）
7. **APIキーは最小権限のサービスアカウント**を使用
8. **本番環境の認証情報にはアクセスさせない**
9. **定期的にアップデート**を適用

## 6. 代替案の検討

| 選択肢 | メリット | デメリット |
|--------|---------|-----------|
| **OpenClaw 導入（隔離環境）** | 多機能、メッセージング連携 | セキュリティ管理コスト高 |
| **Claude Code のみで運用** | セキュリティが堅牢、deny設定可能 | メッセージング連携は別途構築が必要 |
| **しらたま + Claude Code 併用** | 既に構築済み、制御可能 | OpenClawほどの自律性はない |
| **導入を半年待つ** | セキュリティが成熟するのを待てる | 機会損失 |

### 推奨

現時点では**「しらたま + Claude Code」の現行構成を継続**し、OpenClaw は半年後（2026年9月頃）に再評価することを推奨。理由：

- 脆弱性の発見ペースがまだ速い（安定していない）
- マーケットプレイスの信頼性が低い
- Ryo の事業に必要な機能は現行構成でほぼカバーできている
- 導入のセキュリティ管理コストが現在のリソースに対して高すぎる
