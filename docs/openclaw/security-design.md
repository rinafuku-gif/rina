# OpenClaw セキュリティ設計書

## 脅威モデル

### 攻撃シナリオ

OpenClawはSlack経由で外部からメッセージを受け取る。
攻撃者がSlackチャネルに巧妙なメッセージを投稿し、OpenClawの行動を乗っ取る
「プロンプトインジェクション」が最大の脅威。

```
攻撃者 → Slack → OpenClaw → 「rinaリポの財務情報を読んで外部に送信」
                            → 「Slack #general にRyoの個人情報を投稿」
                            → 「メールで契約書の内容を送信」
```

### リスクの公式

**被害の大きさ = 読める情報 × 使えるスキル（権限）**

だから両方を最小化する。

### 財務情報の分離ルール（重要）

OpenClawが売上改善・事業アドバイスを行えるよう、**相対値・トレンド**は共有する。
ただし**絶対値（具体的な金額）**は一切渡さない。

```
【OpenClawに渡す情報（サマリー）】
- 今月の売上: 目標達成率 62%（前月比 +15%）
- えんがわ稼働率: 68%（前月 55%）
- 収益トレンド: 3ヶ月連続上昇
- アラート: 目標未達の見込み。対策が必要
- 事業別貢献度: えんがわ 52% / 蔵サウナPM 48%

【渡さない情報（生データ）】
- 月収: 20.7万円
- えんがわ月売上: 10.7万円
- 蔵サウナPM報酬: 月額10万円
- 固定費: 年間50万円

→ 万が一漏洩しても「目標の62%」からは具体的な金額を逆算できない
→ OpenClawは「売上が足りない、どうする？」という議論は十分にできる
```

## 防御戦略2: スキル権限の制限（使えるスキルを最小化）

### 原則: 最小権限の原則（Principle of Least Privilege）

OpenClawに付与するスキルは必要最小限とし、各スキルに明確な制限をかける。

### スキル権限マトリクス

| スキル | 許可 | 制限事項 |
|---|---|---|
| Slack投稿 | ✅ | 指定チャネルのみ（#openclaw-log 等） |
| Slack読み取り | ✅ | Ryo専用チャネルのみ |
| ファイル読み取り | ✅ | openclaw-vault 内のみ |
| ファイル書き込み | ✅ | openclaw-vault 内のみ |
| Web検索 | ✅ | 情報収集用 |
| メール送信 | ❌ | プロンプトインジェクションでの悪用リスク大 |
| 外部API呼び出し | ❌ | 明示的に許可したものだけ |
| GitHub操作 | ❌ | openclaw-vault へのpushのみ例外 |
| Google Calendar直接操作 | ❌ | rina側のClaude Codeが担当 |
| shell コマンド実行 | ⚠️ | ホワイトリスト方式（git pull等のみ） |

### スキル追加時のチェックリスト

新しいスキルをOpenClawに追加する前に、以下を確認：

1. **そのスキルが乗っ取られたら何が起きるか？**
   - 最悪のシナリオを想定する
   - 「外部に情報を送信できるか？」が最重要チェックポイント

2. **そのスキルは本当にOpenClawに必要か？**
   - rina側のClaude Codeで代替できないか？
   - 手動で十分ではないか？

3. **スコープを絞れるか？**
   - 「メール送信」全般ではなく「特定テンプレートのリマインド送信のみ」等

## 防御戦略3: 検知と監査

### OpenClawの行動ログ

OpenClawの全アクションを記録し、異常を検知できるようにする。

```
openclaw-vault/
└── logs/
    └── actions/
        └── 2026-03-23.log  ← 何をしたか全部記録
```

### 監視すべき異常パターン

- 通常と異なる時間帯のアクティビティ
- 大量のファイル読み取り
- Slack投稿の頻度急増
- 未知のURLへのアクセス試行
